# duoVoiceCoach — Phases 10–12 Design Spec

**Date:** 2026-04-22  
**Status:** Approved — ready for implementation planning  
**Sequence:** Phase 10 → Phase 11 → Phase 12 (sequential; each phase gates the next)

---

## Context

Phases 0–6 delivered a working desktop Spanish voice coach (mic → Whisper → Claude → ElevenLabs TTS). Phase 7 adds Android/PWA support. Phases 8–9 cover cloud deployment and Windows packaging.

Phases 10–12 are the next evolution layer: disciplined code review, a layout redesign, and meaningful feature expansion. They are ordered so that refactoring precedes GUI work (avoiding double-touching files) and feature expansion follows a clean, well-structured codebase.

---

## Phase 10 — Code Review & Refactor

### Goal

Systematic review of all backend and frontend code after six phases of rapid delivery. Identify and resolve complexity hotspots, efficiency gaps, and test coverage holes before the codebase grows further. This is a discipline checkpoint, not a response to a specific bug.

### Scope

**Backend**
- `main.py` — verify routes contain no business logic; all logic delegated to `coach.py`, `session.py`, or provider modules
- `coach.py` — review system prompt construction, mode routing, and correction trigger detection for clarity and single responsibility
- `session.py` — review data model, persistence logic, and serialization for correctness and efficiency
- `stt.py` / `tts.py` — verify abstraction boundaries are clean; no provider-specific logic leaking into the base
- `ai/claude.py` — review prompt caching usage, structured output parsing, and error handling path

**Frontend**
- `useVoice.js` — review state machine correctness, AudioContext lifecycle, and error surface
- `App.jsx` — check for state that belongs in the hook vs. component; verify session config flow
- All components — check for unused props, missing test coverage, and CSS not referenced by any component

**Tests**
- Measure backend coverage; identify untested branches in `coach.py` and `session.py`
- Verify integration test still exercises the full WAV → `/turn` → JSON pipeline
- Flag any tests that mock where they should use real implementations (per existing project rule)

### Output

1. Refactor report written to `docs/superpowers/specs/YYYY-MM-DD-phase10-refactor-report.md` — findings listed with severity: **fix now** / **defer** / **accept as-is**
2. Implementation of all **fix now** findings
3. No speculative refactors — only what the review surfaces

### Gate

- All existing tests pass after refactor (no regressions)
- Refactor report committed and reviewed
- Manual smoke test: full voice session still works end-to-end

## Phase 11 — GUI Layout Redesign

### Goal

Restructure the app from its current single-column layout into a two-pane desktop layout. The left pane is a full-height conversation view with the voice button pinned to the bottom (chat-app pattern). The right pane stacks session config (collapsible), corrections overlay, and session history.

### Selected Layout: Option B — Chat + Tools Split

```
┌─────────────────────────────┬──────────────────────┐
│                             │  ▸ Session Config    │
│  Transcript                 │  ──────────────────  │
│  (scrollable, full height)  │  Corrections         │
│                             │  ──────────────────  │
│                             │  Session History     │
│  [   🎤 Voice Button   ]   │                      │
└─────────────────────────────┴──────────────────────┘
```

**Left pane (~65% width):**
- Full-height scrollable transcript (bilingual, user right / coach left)
- Voice button pinned to bottom center — always visible, never scrolls away
- Error state shown inline below voice button

**Right pane (~35% width):**
- Session config: collapsible `<details>` block — collapsed by default during active conversation
- Corrections overlay: appears below config when corrections are present; auto-clears after 8 seconds or on next turn
- Session history: list of past sessions; tap to load

**Responsive behavior:**
- Below 768px: right pane collapses into a bottom sheet / drawer (preserves Android PWA usability from Phase 7)
- Above 768px: two-pane layout as described

### Components Affected

| Component | Change |
|-----------|--------|
| `App.jsx` | New two-column layout wrapper; right-pane composition |
| `Transcript.jsx` | Full-height flex container; user/coach bubble alignment |
| `VoiceButton.jsx` | Pinned bottom position within left pane |
| `SessionConfig.jsx` | Wrapped in collapsible `<details>`; collapsed default |
| `CoachOverlay.jsx` | Moved into right pane; auto-dismiss timer |
| `SessionHistory.jsx` | Moved into right pane; no structural change |
| `App.css` / `index.css` | New layout grid; responsive breakpoint at 768px |

### What Does NOT Change

- All existing component logic and props interfaces — layout only
- Backend API — no changes
- Existing tests — all should pass unchanged; add layout-level Vitest snapshots

### Gate

- All existing tests pass
- Two-pane layout renders correctly on desktop
- Right pane collapses to drawer on mobile (manual test on Android)
- Manual smoke test: full session in new layout

## Phase 12 — Feature Expansion

### Goal

Add meaningful depth to the coaching experience across three workstreams. Each workstream gets its own spec before implementation begins. Phase 12 is a planning + initial implementation phase.

### Workstream 1: Progress Tracking

The app already stores every `Turn`, `Correction`, and session in JSON. Progress tracking surfaces what is already captured.

**Features:**
- Correction frequency breakdown by type: article usage, verb conjugation, vocabulary, pronunciation (based on `Correction.explanation` text or a new `category` field on `Correction`)
- "Words I keep getting wrong" list — corrections for the same `original` across multiple sessions
- Session frequency view — simple streak or sessions-per-week display
- Visible in a new **Progress** tab or expandable section within session history

**Data model change:** Add optional `category: str` field to `Correction` — e.g., `"article"`, `"conjugation"`, `"vocabulary"`, `"other"`. Claude populates this via structured output. Existing sessions without it degrade gracefully.

### Workstream 2: Structured Lessons

Alongside freeform conversation, offer topic-specific guided exchanges before opening to freeform.

**Mechanics:**
- New `session_mode` value: `"lesson"` (alongside existing `"conversation"`)
- Lesson mode drives the system prompt with an ordered sequence of prompts for the selected topic and level — e.g., for "ordering food" at level 5: `["Greet the waiter", "Ask for the menu", "Order a dish", "Ask for the bill"]`
- Coach steps through each prompt in sequence, confirms completion, then transitions to freeform
- Lesson sequences stored as JSON config in `backend/lessons/` — one file per topic, keyed by level band

**New backend components:**
- `backend/lessons/` directory with topic JSON files
- `GET /lessons/{topic}` route — returns lesson sequence for a topic at a given level
- `backend/coach.py` — new `lesson_step` tracking in session state

**Frontend:** Lesson mode shows current step indicator in right pane (e.g., "Step 2 of 4: Ask for the menu")

### Workstream 3: Open Brainstorm

Items worth exploring — each requires its own mini-spec before any implementation:

- **Vocabulary flashcard review** — between sessions, review words from logged corrections in a simple Q&A format (no new UI framework; React state only)
- **Coach personality modes** — `"formal_tutor"` vs. `"friendly_partner"` — a system prompt variation, not a new provider
- **Correction export** — download session corrections as CSV or Anki-compatible format (`backend/export.py`)

These are candidates, not commitments. Each gets evaluated and specced when the other two workstreams are delivered.

### Gate

- Workstream specs written before any workstream implementation begins
- Progress tracking: correction category field added and populated by Claude; stats view renders
- Structured lessons: lesson sequences work for at least 3 topics at 2 level bands; manual smoke test
- Open brainstorm: at least one item specced and implemented
